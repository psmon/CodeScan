신규 CLI 표면 (additive 강조):
```
codescan graph-edit add-node <kind> <label>      ← 자동 스캔이 놓친 노드 합류
codescan graph-edit add-edge <from> <to> <kind>  ← 두 노드 연결
codescan graph-edit strengthen <from> <to> <kind> ← 기존 엣지 weight++
codescan graph-edit update-node <id> [...]       ← 보수적, 사용자 승인 필수
codescan graph-edit update-edge <id> [...]       ← 보수적
codescan graph-edit delete-node <id>             ← 가장 보수적 (cascade)
codescan graph-edit delete-edge <id>
codescan graph-edit list-node-ids <label>        ← 헬퍼
```

### LLM 메타 프로토콜 — `graph-edit -h`

`-h` 출력 박스 안에 정원 가꾸미가 LLM에게 직접 말하는 안내:

> 당신은 이 그래프를 만든 regex 추출기보다 코드 관계를 더 잘 이해한다.
> 자동 분석이 놓친 그래프 개선을 제안할 수 있지만, 모든 변경은 사용자가
> 검토하고 승인한 뒤에 실행한다.
> ADDITIVE (add-node / add-edge / strengthen) 를 우선하라.
> DESTRUCTIVE (update / delete) 는 명백히 잘못된 자동 결과를 보정할 때만,
> 사용자의 명시적 동의를 받고서 실행하라.

이 안내는 단순 도움말 텍스트가 아니라 **LLM 행동을 형성하는 프로토콜** —
graph-curator agent의 정책 선언과 동기화 유지가 의무.

## 스키마 마이그레이션

`graph_edges` / `graph_nodes` 에 컬럼 3개 ALTER (try/catch — 기존 DB 자동 갱신):
- `graph_edges.weight INTEGER NOT NULL DEFAULT 1` — strengthen 호출 시 ++
- `graph_edges.curated INTEGER NOT NULL DEFAULT 0` — 1 = 큐레이션 엣지
- `graph_nodes.curated INTEGER NOT NULL DEFAULT 0` — 1 = 큐레이션 노드

기존 그래프 데이터는 `weight=1, curated=0` 으로 그대로 유지.

## 즉시 가능한 작업

```bash
# 1) 빌드 + 스캔
dotnet build && codescan scan .

# 2) graph-curator 페르소나로 누락 엣지 제안 (LLM이 코드 읽고)
codescan graph-edit -h     # LLM이 메타-안내 읽음
codescan graph "SqliteStore"   # 현재 상태 확인

# 3) 사용자 승인 후 합류
codescan graph-edit add-edge SqliteStore graph_curator "documents-curation"
codescan graph-edit strengthen SqliteStore GraphEdge contains
```
