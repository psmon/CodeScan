## Source of truth

All channels point at the same GitHub Release assets generated by
`.github/workflows/release.yml`:

```
codescan-win-x64.zip
codescan-linux-x64.tar.gz
codescan-linux-arm64.tar.gz
codescan-osx-x64.tar.gz
codescan-osx-arm64.tar.gz
checksums.txt
sbom.cdx.json
```

Per-asset SHA256s live in `checksums.txt`. Each channel's "update" step pulls
SHA256s from there and rewrites the corresponding manifest template.

## v1 release procedure

1. **Cut a release**
   - Tag: `git tag v<x.y.z>; git push --tags`
   - Or run `release.yml` manually via `workflow_dispatch`.
   - CI builds the five matrix targets, generates SBOM, aggregates
     `checksums.txt`, and publishes a GitHub Release.

2. **Verify the release** вЂ” download one asset, run it, confirm
   `codescan --version` matches.

3. **Update channels (manual in v1)**
   - **winget**: see `packaging/winget/README.md`
   - **Homebrew**: see `packaging/homebrew/README.md`
   - **npm**:
     ```bash
     cd packaging/npm/codescan-cli
     npm version --no-git-tag-version <x.y.z>
     npm publish --access public
     ```

## v1.x automation

`.github/workflows/release.yml` includes three `if: false`-gated publisher jobs
(`publish-winget`, `publish-homebrew`, `publish-npm`). Enable them one channel
at a time after the manual flow has succeeded at least once and credentials
have been added to repo secrets:

| Job | Required secret |
|-----|-----------------|
| publish-winget | `WINGET_TOKEN` (PAT with PR rights on the `microsoft/winget-pkgs` fork) |
| publish-homebrew | `HOMEBREW_TAP_TOKEN` (PAT with push rights on `psmon/homebrew-codescan`) |
| publish-npm | `NPM_TOKEN` (npm automation token) |
